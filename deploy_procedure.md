# 🚀 통합 플랫폼(Multiple Projects Portfolio) 배포 및 업데이트 표준 절차

> **프로젝트 범위 및 주의 사항:**
> 본 저장소는 **Next.js 기반의 멀티 프로젝트 통합 플랫폼(Cover Battle Arena, 학술 논문 아카이브 및 소설 시각화 플랫폼 등)**입니다. 본 저장소에는 코드가 직접 보관된 내부 프로젝트와 링크로만 연결된 외부 프로젝트가 공존합니다. 배포 진행 시 다음 절차를 엄격히 준수하여 무결한 릴리즈를 보장하십시오.

---

## 1. 프로젝트별 사전 데이터 컴파일 & 아웃링크 검증

업데이트 대상 프로젝트의 특성(내부 코드 보유 vs 외부 링크 연결)에 맞춰 아래 사전 검증 단계를 수행하십시오.

### 🔬 [내부 프로젝트 1] 학술 논문 (Spatial Vibration Hypothesis)
* **논문 마크다운 컴파일:** 논문 텍스트(`projects/notion_papers/`)를 수정했거나 신규 개정판을 작성했다면, 커밋 전에 반드시 컴파일러 스크립트를 실행하여 정적 TS 데이터를 갱신하십시오.
  ```bash
  node scripts/compile-papers.js
  ```
* **데이터 검증:** `components/PaperPlatform/paperData.ts`에 번역본 및 개정 버전 단락이 누락 없이 컴파일되었는지 확인하십시오.

### 🎵 [내부 프로젝트 2] Cover Battle Arena (K-pop 투표 플랫폼)
* **API 및 데이터 검증:** 본 저장소에 구현된 API 라우트(`pages/api/vote-video`, `pages/api/videos`) 및 유튜브 투표 채널 데이터의 정합성을 검토하십시오.

### 🗺 [내부 프로젝트 3] 소설 시각화 플랫폼 (눈물을 마시는 새 & 삼체)
* **D3.js 데이터셋 검증:** 4대 선민종족 지형 좌표 데이터 및 인물 관계도용 D3 JSON 데이터 파일의 마크업 정합성을 사전 점검하십시오.

### ⚖️ [외부 프로젝트 1] BLL Pro & 📈 [외부 프로젝트 2] KyyQuant
* **외부 독립 프로젝트 연결성 검증:**
  * **BLL Pro**와 **KyyQuant**의 실제 개발 코드는 본 저장소에 존재하지 않으며, 완전히 독립된 별개의 원격 저장소 및 서버 파이프라인에서 배포와 업데이트를 처리합니다.
  * 따라서 본 플랫폼을 배포하기 전에는 오직 메인 랜딩페이지 데이터(`shared/data/projects.ts`)에 정의된 이들의 **외부 아웃링크 URL** 및 **소개 정보(개발 완료율, 데이터 건수 등의 메트릭)**가 최신 정보로 올바르게 명시되어 있는지 정합성 검사만 수행하십시오.
    * BLL Pro 주소: `https://www.bll-pro.com/`
    * KyyQuant 주소: `https://kyyquant-ai-solution.vercel.app/`

## 2. 공통 로컬 빌드 테스트 (사전 컴파일 검증 ⭐️필수)
* 임의의 내부 코드나 데이터를 수정한 후, 원격 저장소에 푸시하기 전 반드시 로컬에서 전체 프론트엔드 통합 빌드 테스트를 실행하십시오.
  ```bash
  npm run build
  ```
* 사소한 타입 에러나 린트 경고만으로도 전체 플랫폼 Vercel 배포가 중단(Error)될 수 있으므로, 푸시 전 로컬 빌드 성공 확인은 필수 프로세스입니다.

---

## 3. 로컬 포트 3000 및 개발 환경 락 관리
* 로컬 개발 서버(`npm run dev`) 가동 시, 포트 3000 선점이나 Next.js 내부 개발 락(`.next/dev/lock`) 획득 실패 오류가 발생하는 경우 아래 명령어로 환경을 정규화하십시오.
  ```bash
  # 포트 3000 점유 중인 프로세스 강제 종료
  lsof -i :3000
  kill -9 <PID>

  # NextJS 개발 서버 락 강제 소거
  rm -f .next/dev/lock
  ```

## 4. 데이터베이스 및 Supabase 동기화
* 서비스 운영에 공유되는 Supabase PostgreSQL 테이블 수정이 동반되는 경우, 코드 푸시 전에 Supabase SQL Editor를 통해 쿼리를 적용하여 데이터 정합성을 먼저 동기화하십시오.

## 5. 버전 관리 및 브랜치 커밋 규칙
* **전체 스테이징:** 변경이 완료되면 누락 파일이 없도록 전체 추가 후 의미 있는 커밋 메시지로 기록합니다.
  ```bash
  git add .
  git commit -m "feat([Project/Area Name]): [Summary of changes]"
  ```
* **브랜치 병합:** 개별 브랜치에서 작업한 경우 `origin` 원격지로만 커밋을 푸시하며, `main` 브랜치로의 직접 병합은 **반드시 프로젝트 관리자(사용자)의 최종 승인**을 거친 후에 실행하십시오.

---

## 6. 원격 저장소 푸시 및 자동 배포 (CI/CD)
* **통합 웹 Portal (Vercel):**
  * `main` 브랜치에 코드가 푸시되면 Vercel.com 에서 자동으로 감지하여 최적화된 프로덕션 번들을 약 1~2분 내에 빌드 및 글로벌 배포합니다.
  * 내부 프로젝트(논문 플랫폼, Cover Arena, 소설 시각화 등)의 업데이트 사항은 프론트엔드 빌드 시 통합 반영되어 라이브 서비스에 자동 배포됩니다.

---

## 7. 최종 배포 상태 검수 및 모니터링
1. **Vercel 대시보드 검수:** Vercel 배포 탭에서 빌드가 `Ready` 상태로 정상 완료되었는지 파악하십시오.
2. **실제 라이브 서비스 테스트:** 모바일 및 브라우저 환경에서 메인 랜딩페이지(`http://localhost:3000` / 도메인)에 접속하여 각 서브 프로젝트 바로가기 배너 연동이 정상 작동하는지 전수 검수하십시오.
